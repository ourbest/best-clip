# 视频抽取与切片规则说明

本文档用于对齐 Python 端现有的视频抽取和选择逻辑，作为 `mobile` 项目后续逐步实现的依据。

目标不是一次性把所有能力搬完，而是先把规则说清楚，再按文档分阶段落地到 iOS 项目中。

## 1. Python 端当前流程

Python 端的完整链路可以拆成 4 步：

1. `step2_timeline.py`
   - 以字幕块 `chunk_index` 为单位做时间线抽取。
   - 每个 chunk 单独调用 LLM，把大纲和 SRT 文本映射成带起止时间的切片。
   - 最终把所有 chunk 的结果按 `start_time` 全局排序，并重新分配固定 `id`。

2. `step3_scoring.py`
   - 对每个切片做内容评分。
   - 输入给 LLM 的核心字段是：
     - `outline`
     - `content`
     - `start_time`
     - `end_time`
   - LLM 返回：
     - `final_score`
     - `recommend_reason`
   - 评分后按阈值 `MIN_SCORE_THRESHOLD` 筛出高分切片。

3. `step4_title.py`
   - 只对高分切片生成标题。
   - 输入仍然按块分组，减少一次性请求规模。
   - 若标题生成失败，保留原始片段，避免数据丢失。

4. `step5_clustering.py`
   - 对带标题的高分切片做主题聚类。
   - 先做关键词预聚类，再调用 LLM 聚类。
   - LLM 失败时回退到：
     - 关键词预聚类结果
     - 默认高分/中分分组

## 2. Python 端的切片和选择逻辑

### 2.1 时间线抽取逻辑

- 按 `chunk_index` 处理，不跨块混算。
- 每个 chunk 都会读取对应的 SRT 块文件。
- 输出的时间区间必须落在 chunk 的起止时间范围内。
- 解析失败会重试，并保存原始响应用于排查。

### 2.2 评分逻辑

- 评分对象是切片，不是整段视频。
- 评分依据主要是内容质量和可用性。
- 评分结果同时包含：
  - 分数
  - 推荐理由
- 评分后按阈值过滤，只有高分片段继续进入后续步骤。

### 2.3 标题逻辑

- 只有高分片段才会进入标题生成。
- 标题生成按 chunk 批量进行。
- 生成失败时保留原始片段，避免中断后续流程。

### 2.4 聚类逻辑

- 聚类目标是把高分片段按主题组织成合集。
- 逻辑顺序是：
  - 关键词预聚类
  - LLM 聚类
  - LLM 失败后的默认分组
- 每个合集限制最大片段数量。

## 3. 对应到 iOS / mobile 的实现原则

mobile 端不直接照搬 Python 的文件结构，而是照搬它的选择顺序和兜底方式。

### 3.1 先抽取，再评分

先把素材变成可比较的“抽取单元”：

- 照片
- 视频片段
- 视频的多个子片段

然后再对这些单元做评分和筛选。

### 3.2 先内容筛选，再标题和聚类

只有入选的高分单元才继续进入：

- 标题生成
- 主题归类
- 最终成片编排

### 3.3 失败要能回退

任何 LLM 步骤失败时，都必须能回退到本地规则：

- 评分失败时回退到本地打分
- 标题失败时保留原始文本
- 聚类失败时回退到关键词或默认分组

## 4. mobile 端建议的分步实现顺序

### 第 1 步：视频抽取单元化

先把视频处理拆成明确的抽取单元：

- `VideoAssetAnalyzer`
- `MotionSegmentPolicy`
- `ContentSegmentPolicy`
- `VideoSegmentScorer`

目标：

- 让视频不只是“一个源文件”，而是可以被评分和切片的一组候选段。

### 第 2 步：本地评分和筛选

把 Python 的 `step3_scoring.py` 思路搬到 iOS：

- 为每个素材计算分数
- 按阈值筛掉低质量内容
- 保留推荐理由，供 UI 和后续步骤展示

### 第 3 步：标题生成

把 Python 的 `step4_title.py` 思路搬到 iOS：

- 只对高分素材生成标题
- 生成失败时保留原始文案
- 标题生成结果进入结果页和聚类步骤

### 第 4 步：主题聚类

把 Python 的 `step5_clustering.py` 思路搬到 iOS：

- 先按关键词做预分组
- 再让 LLM 做主题聚类
- 最后给出合集结构和兜底集合

## 5. 推荐的 mobile 模块边界

建议把规则拆成下面几个模块：

- `VideoAssetAnalyzer`
  - 负责多帧视频分析

- `MotionSegmentPolicy`
  - 负责“切几段”的判断

- `ContentSegmentPolicy`
  - 负责“每段多长”的判断

- `VideoSegmentScorer`
  - 负责把运动和内容规则组合成可执行的切片结果

- `AssetSummaryBuilder`
  - 负责把素材汇总成评分输入

- `CompositionPlanner`
  - 负责把评分结果和推荐结果转成最终 `CompositionPlan`

## 6. 逐步落地时的验收标准

每一步都要满足下面的最小标准：

1. 规则能被单元测试覆盖。
2. LLM 失败时能回退到本地逻辑。
3. 新规则不会破坏现有导入、预览、导出流程。
4. 视频的“内容丰富”和“运动强度”都能影响最终成片。

## 7. 当前状态

当前 `mobile` 项目已经具备以下基础能力：

- 视频多帧分析
- 视频稳定度 / 运动度建模
- 视频切片评分器雏形
- 成片拼接按区间裁切

下一步的实现重点就是把这些规则和 Python 端的选择逻辑持续对齐。
